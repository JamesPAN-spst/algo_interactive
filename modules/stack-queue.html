⚙️ 前置知识

线性表操作 时间线倒流

栈（LIFO）与队列（FIFO）本质上是被“封印”了随机访问能力的数组。它们存在的意义不是为了灵活，而是为了在特定的出入顺序下（如历史回退、广度扩散）提供极致简单的 O(1) 状态管理。学会这两种数据结构，就是学会了如何掌控程序的“时间沙漏”。

💡 推导：特殊栈与特殊队列的诞生

1. 最小栈 (Min-Stack) —— 为什么不开两个指针找最小值？

全局一个最小值 (Naive)

因为如果当前最小的元素（比如 1）被 Pop 出栈了，栈里仅次于它的“第二小”元素（比如 3）没人记得！你需要 O(N) 遍历重新找。这违背了栈 O(1) 的初衷。

伴生备忘录 (Smart)

我们在向栈内压入数据的时候，顺便压入一个元组 (val, 当前最小)。每个元素都像带着一个紧箍咒一样记住了它进栈“那一刻”的全局最小值！它死了哪怕第二小元素漏了，第二小元素的紧箍咒也会瞬间恢复之前的最小值。

2. 单调栈 (Monotonic Stack) —— 寻找“下一个更大元素”

1
破而后立：如果我正在维护一个递减栈 [10, 8, 5]，此时来了一个庞然大物 9。它破坏了递减性！
2
榨干剩余价值：9 嚣张地要把之前的 5 和 8 挤走。但在它们滚蛋（Pop）之前，我们恍然大悟：把他们挤走的人，不正是它们苦苦寻找的右侧“下一个更大元素”吗？！ 于是我们记录 5的右霸=9，8的右霸=9，然后 9 坐稳了它的位置 [10, 9]。
Q: 为什么单调栈能把寻找所有 Next Greater 的死循环降低到了 O(N)？
查看答疑 因为在这个暴力的“弱肉强食”淘汰赛中，虽然新来的元素可能一下子踢出去了好几个较小的元素（看作多层循环）。但是，所有在数组里的元素，一生都只进栈一次，出栈一次！不论你怎么循环，总的进出次数严格限定在 2N 之内。这就是均摊分析出的极限 O(N)。

3. 滑动窗口最大值与单调队列 (Monotonic Queue)

这本质上是一个有寿命限制的“选秀节目”。如果一个选手 x 来了，发现前面有个选手 y 年纪比他大（位置更靠前，死得早）还比他弱（数值小）：那 y 就连一点残存价值都没有了（它不可能成为最大值了），直接从队尾踢掉！
然而，为什么不能用单调栈？因为栈只能在一头操作，而这里活得太老的选手（寿命到了逃出窗口）需要从队头 popleft() 离开！所以必须使用双端队列 (Deque)。

💻 模板代码：单调栈与双端单调队列

# 1. 单调栈找 Next Greater Element
def next_greater(nums):
    res = [-1] * len(nums)
    stack = [] # 里面存放的是索引 (index)
    
    for i in range(len(nums)):
        # 当遇到高个子，把栈里不如它高的直接踢了，并结算它们的答案
        while stack and nums[i] > nums[stack[-1]]:
            popped_idx = stack.pop()
            res[popped_idx] = nums[i]
        stack.append(i) # 当前小弟老老实实进栈排队
    return res

# 2. 单调队列：滑动窗口的最大值
from collections import deque
def max_sliding_window(nums, k):
    res = []
    dq = deque() # 单调递减队列，存放索引
    
    for i in range(len(nums)):
        # 1. 淘汰太老的队员（如果队头的寿命超出了窗口左边界）
        if dq and dq[0] < i - k + 1:
            dq.popleft()
            
        # 2. 从队尾把不如自己大、不仅老还菜的全踢了！
        while dq and nums[i] >= nums[dq[-1]]:
            dq.pop()
            
        # 3. 自己成为新一届的尾巴入队
        dq.append(i)
        
        # 什么时候记录答案？当窗口完全成型时。
        if i >= k - 1:
            res.append(nums[dq[0]])  # 当前窗口毫无疑问最大的是谁？队头！
    
    return res
技巧时间空间为什么
找单一侧的极大/极小（如雨水问题）O(N)O(N)单调栈。利用 O(N) 的历史存储记录单侧遮挡。
滑动窗口内极值O(N)O(K)双端队列。除了被踢出去的，最大维持 K 个候补。

🔴 易错点分析

❌ 栈内存存错东西

# 将数值存入栈中
stack.append(nums[i])
# ... 等你弹出来的时候，你根本不知道这个弹出来的小数字
# 它的本尊在原数组什么位置，你怎么把结果写进答案数组 `res` 里？

✅ 单调栈绝大多数情况必须存索引

# 存入索引。
stack.append(i)
# 到时候根据索引拿值 `nums[popped_idx]`，并且能写进 `res[popped_idx]`。

🔷 竞赛拓展

单调队列优化 DP： 滑动窗口不仅仅是求最大值。如果在动态规划中，状态转移方程长成 `dp[i] = max(dp[j]) (i-k <= j < i)`，那么我们如果用 $O(K)$ 循环直接查找必死无疑。只要使用单调队列维护这个“长度有保质期”的 DP 窗口最大值，立刻可以将 O(NK) 算法降维打击到 O(N)！这是极为恐怖的性能跨越。
