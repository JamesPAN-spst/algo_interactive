知识点总结提纲（Knowledge / Connaissances）

🎯 引子问题

"手机输入法打 'hel'，怎么瞬间弹出 'hello', 'help', 'helmet'？"

Hash 表可以告诉你某个完整单词是否存在，但怎么高效找出所有以某个前缀开头的单词？这就是字典树（Trie）要解决的核心难题。

⚙️ 前置知识

树的节点指针 散列表 (HashMap) 字符串基础

树结构：Trie 是多叉树——每个节点可以有多个子节点（最多 |Σ| 个，Σ 为字母表）。

HashMap：提供 O(1) 的精确查找，但无法高效回答"以某前缀开头的所有键"——理解这一局限是学习 Trie 的起点。

🔬 研究动机——HashMap 为什么搞不定前缀查询？

HashMap（精确搜索之王）

查 "apple" 是否在字典中？Hash("apple") → O(L)。✅
查以 "app" 为前缀的所有 100 个单词？只能 O(N) 遍历全部 10 万条记录逐个比较前缀。❌
问题：哈希函数把不同前缀的单词打散到不同桶里，完全丢失了前缀的结构信息。

Trie（前缀之王）

"apple" 在 Trie 中是路径 根→a→p→p→l→e。
查以 "app" 开头的词？沿路径走三步到达 'p' 节点，其下整棵子树都是 "app" 前缀的词汇！
前缀搜索 = 沿路径走到对应节点 + DFS 收集子树。

形式化定义：给定 $n$ 个总长为 $S$ 的字符串，设计数据结构支持：① insert(word) $O(L)$，② search(word) $O(L)$，③ startsWith(prefix) $O(L)$，其中 $L$ 为查询串长度，与字典大小 $n$ 无关。

💡 推导：Trie 的设计与压缩

1. 基本 Trie 结构

1
核心想法：把每个字符串拆为逐字符的路径。共享相同前缀的字符串共用同一段路径，分歧点才分叉。
2
节点结构：每个节点存储 children（字符→子节点的映射）和 is_end（标记是否为完整单词的终止）。
插入 "cat", "car", "card", "dog" 后的 Trie:

         (root)
        ╱      ╲
      [c]       [d]
       │         │
      [a]       [o]
      ╱ ╲        │
    [t●] [r●]   [g●]
          │
         [d●]

  ● = is_end = True（完整单词终止符）
Q: 树里有 "app" 和 "apple"，如果不加 is_end 标记，怎么区分 "app" 是真实单词还是仅仅是 "apple" 的中转站？
查看答疑 这就是 is_end 存在的终极意义！在 'p' 节点（第二个 'p'）上设 is_end = True，宣告"从根到我的路径 app 本身是一个合法单词"——即使 'p' 之后还有子节点 l → e 通向 "apple"。

2. Trie 的前缀共享与内存复用

核心洞察：如果字典中有 10 万个词都以 "ab" 开头（abandon, absolute, abstract...），HashMap 里每个词都独立存储 'a','b' 两个字符。Trie 里它们共享同一条 根→a→b 通道——前缀越共享，Trie 越省空间。

3. 压缩 Trie (Radix Tree / Patricia Tree)

1
普通 Trie 的浪费：如果单词 "internationalization" 只有它自己用这段路径，就需要 20 个单字符节点串联，浪费空间。
2
压缩思路：将只有单个子节点的连续链合并为一个节点，存储"子串"而非"单字符"。节点数从 O(S) 降为 O(n)（n = 单词数量）。
普通 Trie:               压缩 Trie (Radix Tree):

    (root)                   (root)
     │                      ╱      ╲
    [r]                  ["rom"]  ["rub"]
     │                    ╱  ╲       ╱ ╲
    [o]              ["an●"]["e●"]["ber●"]["y●"]
     │               │
    [m]           ["ce●"]
    ╱ ╲
  [a]  [e●]      左：存 roman, romance, rome
   │              右：存 rubber, ruby
  [n]
   ╱ ╲
 [●]  [c]
       │
      [e●]

💻 模板代码

Python

class TrieNode:
    def __init__(self) -> None:
        self.children: dict[str, 'TrieNode'] = {}  # [why] HashMap 比 [None]*26 更灵活，支持任意字符
        self.is_end: bool = False                   # [why] 标记完整单词终止

class Trie:
    def __init__(self) -> None:
        self.root = TrieNode()

    # ─── 插入 O(L) ───
    def insert(self, word: str) -> None:
        node = self.root
        for ch in word:
            if ch not in node.children:
                node.children[ch] = TrieNode()  # [why] 没路则铺路
            node = node.children[ch]
        node.is_end = True  # [why] 盖公章：此路径对应的前缀是合法单词

    # ─── 精确查找 O(L) ───
    def search(self, word: str) -> bool:
        node = self._walk(word)
        return node is not None and node.is_end  # [why] 走到了 ≠ 是单词

    # ─── 前缀判定 O(L) ───
    def starts_with(self, prefix: str) -> bool:
        return self._walk(prefix) is not None  # [why] 能走完就说明前缀存在

    def _walk(self, s: str) -> 'TrieNode | None':
        """沿字符路径走到目标节点，走不通返回 None。"""
        node = self.root
        for ch in s:
            if ch not in node.children:
                return None
            node = node.children[ch]
        return node

    # ─── 收集所有以 prefix 开头的单词 ───
    def autocomplete(self, prefix: str) -> list[str]:
        node = self._walk(prefix)
        if not node:
            return []
        results: list[str] = []
        def dfs(n: TrieNode, path: str) -> None:
            if n.is_end:
                results.append(path)
            for ch, child in sorted(n.children.items()):
                dfs(child, path + ch)
        dfs(node, prefix)
        return results
C++ 版本 
#include <string>
#include <unordered_map>
#include <vector>

struct TrieNode {
    std::unordered_map<char, TrieNode*> children;
    bool is_end = false;
    ~TrieNode() {
        for (auto& [c, p] : children) delete p; // [why] 避免内存泄漏
    }
};

class Trie {
    TrieNode root;
public:
    void insert(const std::string& word) {
        auto* node = &root;
        for (char ch : word) {
            if (!node->children.count(ch))
                node->children[ch] = new TrieNode();
            node = node->children[ch];
        }
        node->is_end = true;
    }

    bool search(const std::string& word) const {
        auto* node = walk(word);
        return node && node->is_end;
    }

    bool startsWith(const std::string& prefix) const {
        return walk(prefix) != nullptr;
    }

private:
    const TrieNode* walk(const std::string& s) const {
        const auto* node = &root;
        for (char ch : s) {
            auto it = node->children.find(ch);
            if (it == node->children.end()) return nullptr;
            node = it->second;
        }
        return node;
    }
};
Rust 版本 
use std::collections::HashMap;

#[derive(Default)]
struct TrieNode {
    children: HashMap<char, TrieNode>, // [why] HashMap 灵活支持 Unicode
    is_end: bool,
}

#[derive(Default)]
pub struct Trie {
    root: TrieNode,
}

impl Trie {
    pub fn new() -> Self { Self::default() }

    pub fn insert(&mut self, word: &str) {
        let mut node = &mut self.root;
        for ch in word.chars() {
            node = node.children.entry(ch).or_default(); // [why] 没路则默认插入空节点
        }
        node.is_end = true;
    }

    pub fn search(&self, word: &str) -> bool {
        self.walk(word).map_or(false, |n| n.is_end)
    }

    pub fn starts_with(&self, prefix: &str) -> bool {
        self.walk(prefix).is_some()
    }

    fn walk(&self, s: &str) -> Option<&TrieNode> {
        let mut node = &self.root;
        for ch in s.chars() {
            node = node.children.get(&ch)?; // [why] ? 自动返回 None
        }
        Some(node)
    }
}

📊 复杂度分析

操作时间空间为什么
insert(word)O(L)最坏 O(L) 新节点逐字符走/建路径，与字典大小 n 无关
search(word)O(L)O(1)沿路径走 L 步即可判定
startsWith(prefix)O(L)O(1)同上，不需检查 is_end
autocomplete(prefix)O(L + K)O(K)L 步定位前缀节点 + DFS 收集 K 个结果
构建整棵 TrieO(S)O(S · |Σ|)S = 所有字符串总长；每节点最多 |Σ| 个指针

最坏空间注意：若所有单词完全无共同前缀（如 "a", "b", "c"...），退化为每个字符一个节点，空间 O(S)。但实际自然语言中前缀共享率高，Trie 通常比存储所有完整字符串更省空间。

📐 核心性质与定理

1. 前缀共享性
Trie 中任意两个公共前缀只存储一次。
直觉：所有以 "pre" 开头的单词共用 根→p→r→e 这三个节点。

2. 查询独立于字典规模
Trie 的 search/startsWith 时间为 $O(L)$，仅取决于查询串长度 $L$，与字典大小 $n$ 无关。
对比：哈希表虽然也是 $O(L)$ 查找（计算哈希值），但无法做前缀查询。

3. 字典序输出
对 Trie 进行 DFS（按字符排序遍历子节点），可以按字典序输出所有单词。
应用：搜索引擎的自动补全候选按字典序排列。

4. 压缩 Trie 节点数
Radix Tree（压缩 Trie）的内部节点数不超过 $n$（单词数），而非 $S$（总字符数）。
来源：每个内部节点至少有 2 个子节点（否则会被合并）。

基础 实现 Trie (LC 208)

题意：实现 insert, search, startsWith。

思路：直接套上面的模板。关键是 search 必须检查 is_end，startsWith 不需要。

中级 单词搜索 II (LC 212)

题意：在 m×n 字母网格中找出词典里所有单词（可相邻拼接）。

思路：将词典建成 Trie，然后对网格每个格子出发 DFS。DFS 过程中，如果当前路径不是任何词的前缀（Trie 走不通），立即剪枝。比用 HashSet 的优势巨大——HashSet 必须等完整路径形成才能判断，Trie 可以逐字符提前终止。

def findWords(board: list[list[str]], words: list[str]) -> list[str]:
    trie = Trie()
    for w in words:
        trie.insert(w)
    res, m, n = set(), len(board), len(board[0])

    def dfs(i: int, j: int, node: TrieNode, path: str):
        if node.is_end:
            res.add(path)
            node.is_end = False  # [why] 去重：找到后标记防止重复添加
        if i < 0 or i >= m or j < 0 or j >= n:
            return
        ch = board[i][j]
        if ch not in node.children:
            return  # [why] Trie 剪枝：此方向无任何词的前缀
        board[i][j] = '#'  # [why] 标记已访问
        for di, dj in [(0,1),(0,-1),(1,0),(-1,0)]:
            dfs(i+di, j+dj, node.children[ch], path+ch)
        board[i][j] = ch   # [why] 回溯恢复

    for i in range(m):
        for j in range(n):
            dfs(i, j, trie.root, "")
    return list(res)
高级 数组中两个数的最大异或值 (LC 421)

题意：给定整数数组，找出两个数使异或值最大。

思路：01-Trie：将每个数的 32 位二进制表示插入 Trie。查询时，从最高位起贪心地走与当前位相反的方向（1 选 0，0 选 1），尽可能让高位异或为 1。

时间：O(32n) = O(n)，比暴力 O(n²) 快得多。

📋 Trie vs 替代方案对比

数据结构精确查找前缀查询空间优势劣势
HashMapO(L)O(n·L) 全遍历O(S)实现简单、精确查找极快完全不支持前缀操作
排序数组 + 二分O(L·log n)O(L·log n + K)O(S)空间紧凑动态插入 O(n)
TrieO(L)O(L + K)O(S·|Σ|)前缀查询最优空间占用可能大
Radix TreeO(L)O(L + K)O(n)压缩后节点数仅 O(n)实现复杂

S = 总字符数，L = 查询串长，K = 匹配结果数，|Σ| = 字母表大小

1. search 不检查 is_end → WA

❌ 走完路径就返回 True

            return False

    return True  # 错！"app" 只是 "apple" 的中转

✅ 必须检查 is_end

    return node is not None and node.is_end

2. 删除时直接删节点破坏共享前缀

❌ 暴力删除路径上所有节点

# 删除 "app" 时把 a→p→p 全删了
# 但 "apple" 还需要 a→p→p→l→e！

✅ 先清 is_end，再从叶向上逐层删无用节点

# 1. 将 "app" 终点的 is_end 置 False
# 2. 从末尾向上回溯，仅删除无子节点且 is_end=False 的节点
# 3. 遇到有其他子节点或 is_end=True 的节点时停止

3. 固定数组 vs HashMap 子节点

使用 [None]*26 数组时，字符映射为 ord(ch)-ord('a')，效率高但仅支持小写字母。若需要支持 Unicode（如中文），必须用 HashMap。C++ 中注意 unordered_map 的常数较大，竞赛中有时用数组更快。

01-Trie —— 求最大异或值

将所有整数的 32 位二进制表示插入只有 0/1 两个子节点的 Trie。查询时从最高位到最低位贪心走相反方向，O(32) 即可找到与目标异或最大的数。

class BinaryTrie:
    def __init__(self) -> None:
        self.root: list = [None, None]  # [left=0 child, right=1 child]

    def insert(self, num: int) -> None:
        node = self.root
        for i in range(31, -1, -1):  # [why] 从最高位到最低位
            bit = (num >> i) & 1
            if node[bit] is None:
                node[bit] = [None, None]
            node = node[bit]

    def max_xor(self, num: int) -> int:
        node, xor_val = self.root, 0
        for i in range(31, -1, -1):
            bit = (num >> i) & 1
            want = 1 - bit  # [why] 贪心：走相反方向让异或位为 1
            if node[want] is not None:
                xor_val |= (1 << i)
                node = node[want]
            else:
                node = node[bit]
        return xor_val
AC 自动机 (Aho-Corasick) —— 多模式匹配

将多个模式串建成 Trie，然后用 BFS 建立失败指针（failure link）——指向当前节点对应字符串的最长真后缀中存在于 Trie 的节点。匹配时若当前字符走不通，跳转到失败指针继续匹配，时间 $O(|text| + |Σ| \cdot \sum|pattern|)$。这是 Trie + KMP 思想的融合。

后缀自动机 (Suffix Automaton, SAM)

接受一个字符串所有后缀的最小 DFA。节点数 ≤ 2n-1，转移数 ≤ 3n-4。O(n) 构建，可解决大量子串统计问题（本质不同子串数、最长公共子串等）。严格证明涉及等价类理论，此处仅提及。

🏭 工业实战

IP 路由表的最长前缀匹配 (Longest Prefix Match)

网络路由器需要根据目标 IP 地址在路由表中找到最长匹配的前缀来决定转发端口。IP 地址视为比特串，路由表存储在 Trie（或其变体 Patricia Tree）中。查询时沿比特路径走到尽可能深的匹配节点，O(32) 即完成。Linux 内核的 FIB (Forwarding Information Base) 使用 LC-Trie（Level-Compressed Trie）实现。

IDE 自动补全引擎

VS Code、IntelliJ 等 IDE 的代码补全功能底层使用 Trie 或倒排索引。当用户输入 str. 时，引擎在 Trie 中定位 "str." 前缀节点，DFS 收集子树中按使用频率/相关度排序的候选项，实时返回给 UI。TypeScript Language Server (tsserver) 就使用了前缀树加速补全。

搜索引擎 Suggest / Autocomplete

Google 搜索框的自动补全：将数十亿历史查询存入 Trie，每个节点附带频率计数。用户每输入一个字符，系统在 Trie 中定位对应前缀节点，返回频率最高的 Top-K 候选。实际系统会对 Trie 做分区+缓存优化（如 Google 的 Completion Trie），使得延迟低于 100ms。
