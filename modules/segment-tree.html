线段树（Segment Tree）用来解决数组的动态区间连环拷问：我不停地修改数组某个下属、甚至某一段区间的全部数值，并且我不停地问你某个子区间的总和/最大值。你必须同时做到 O(log N) 修改 和 O(log N) 查询！

💡 推导：线段树的由来与 Lazy Tag 的精髓

1. 为什么我们需要线段树？

前缀和数组 (Naive)

提前把前缀累加，查询区间求和 [L,R] = sum[R] - sum[L-1] 是 O(1)！但是如果偷偷修改了一个开头的数字，后面全部倒霉，修改是 O(N)，动态查询会超时。

线段树分治 (Smart)

把数组像大饼一样不断对半切，每块大饼刻着它包含的碎片的汇总统和。修改一个碎片，只会往上报告一侧，影响 $\log N$ 次；查询也最多是由每层的边角料拼接而成，也是 $\log N$。两端平衡！

2. Lazy Tag 懒惰标记推演

1
难题出现：如果我要把 [0, 1000] 的数字全加上 1 （区间修改）。难道我要递归到最底层 1000 个叶子节点一个个加吗？这又变回 O(N) 了。
2
挂机策略 (Lazy)：不！当我发现我的查询/修改区间完美覆盖了当前节点代表的 [0, 1000] 大区间时，我直接修改这个大区间的汇总值（比如 sum += 1000 * 1），然后在它身上贴个红字警告：“注意，我欠你们俩儿子每人 +1”。然后就可以直接 return 下班了！下层的修改完全被“懒惰地”搁置了。
3
清算机制 (Push-Down)：未来的某一天，如果有另一个人来想要详细查询 [0, 500] 的信息，我必须得往深了搜索。这时候我头顶上贴着红字警告的欠款就瞒不住了！在进入儿子节点之前，我必须先Push-Down，把欠款下传给两个儿子，再消除自己的警告红字。
Q: 树状数组 (Fenwick Tree) 与线段树的区别是什么？
查看答疑 树状数组代码极短，常数小，位运算如 `i & -i` 神乎其技。但它能做的东西局限：比如求和求前缀异或可以（因为可以利用减法相消去）。但是它无法处理求区间最大值 Max 的动态修改问题（Max 没有减法逆运算，你不能用 Max(大) 撤销一个元素）。线段树万物皆可维护，甚至能维护区间的一段复杂的哈希信息合并！能用树状数组的大部分都能被线段树降维打击取代。

💻 模板代码：带 Lazy Tag 的线段树 (区间加法、区间求和)

class SegmentTree:
    def __init__(self, data):
        self.n = len(data)
        # 4N 空间是非常稳的经验值，证明与二进制树高满打满算有关
        self.tree = [0] * (4 * self.n)
        self.lazy = [0] * (4 * self.n)
        self.build(data, 0, 0, self.n - 1)

    def build(self, data, node, start, end):
        if start == end: # 叶子节点
            self.tree[node] = data[start]
            return
        mid = (start + end) // 2
        # 左儿子编号 2*i+1, 右儿子 2*i+2 (如果是从1下标开始就是 2*i 和 2*i+1)
        self.build(data, 2 * node + 1, start, mid)
        self.build(data, 2 * node + 2, mid + 1, end)
        self.tree[node] = self.tree[2 * node + 1] + self.tree[2 * node + 2]

    def push_down(self, node, start, end):
        if self.lazy[node] != 0:
            mid = (start + end) // 2
            left = 2 * node + 1
            right = 2 * node + 2
            # 欠款下发
            self.lazy[left] += self.lazy[node]
            self.lazy[right] += self.lazy[node]
            # 根据区间长度更新两边儿子的汇总值
            self.tree[left] += self.lazy[node] * (mid - start + 1)
            self.tree[right] += self.lazy[node] * (end - mid)
            # 钱款两清
            self.lazy[node] = 0

    def update(self, node, start, end, l, r, val):
        if r < start or end < l: return
        
        # 完美覆盖：打标记走人
        if l <= start and end <= r:
            self.tree[node] += (end - start + 1) * val
            self.lazy[node] += val
            return
            
        self.push_down(node, start, end)
        mid = (start + end) // 2
        self.update(2 * node + 1, start, mid, l, r, val)
        self.update(2 * node + 2, mid + 1, end, l, r, val)
        self.tree[node] = self.tree[2 * node + 1] + self.tree[2 * node + 2]
操作时间空间为什么
建树 BuildO(N)O(4N)因为它本质是对整个所有叶子节点进行了一次后序遍历，合并代价为O(1*N)。
单点/区间修改O(log N)-因为完美覆盖时就直接终止，无论多大的区间修改，切割成的小段也超不过 $O(log N)$ 段！
区间查询查询O(log N)-同理，被分割成的拼凑碎快也只有 $log N$ 快。

🔴 易错点分析

❌ 忘记根据区间长度乘上 Lazy 值

# 打标记时
self.tree[node] += val  
# 大错特错：如果你底下的区间有 10 个人，你告诉上面这帮人总和变了多少？
# 变了 10*val！而不是变了 val！求偏导的线性关系必须保持。

✅ 附带权重：时刻认清 node 的身份

# 节点涵盖多宽，数值就翻多少倍
self.tree[node] += (end - start + 1) * val
self.lazy[node] += val

🔷 竞赛拓展

动态开点线段树： 如果你要维护的区间是 $1$ 到 $10^9$ 怎么办？开 $4 imes 10^9$ 的数组内存瞬间爆炸！因为并不是所有节点你都要用到，我们可以“按需分配内存”。使用指针 node.left 和 node.right，只有当访问到时我们才实例化该节点。

主席树 (可持久化线段树)： 在每次 Update 修改节点时，不去修改原节点，而是“复制”一条新的从根到叶子的 $\log N$ 条链并保留旧链。这让你保存了历史所有的版本记录！可以支持查询“第 $t$ 次修改时，区间的情况是什么？”，极其强大。
