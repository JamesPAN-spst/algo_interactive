知识点总结提纲（Knowledge / Connaissances）

⚙️ 前置知识

指针与引用 内存不连续性

链表由于不是连续存储的，因此我们丧失了 O(1) 的随机访问能力，取而代之的是极致的插入/删除灵活性。解链表题的核心心法永远是：管理好你手中的那几根指针，不要让它们弄丢了节点。

💡 推导：双指针的艺术

1. 快慢指针找中点

1
相对速度： 快指针 fast 每次走 2 步，慢指针 slow 每次走 1 步。快指针速度是慢指针的 2 倍。
2
行程等式： 当 fast 走到链表末尾（路程 $L$）时，由于速度是 2 倍，slow 一定刚好走到 $L/2$ 处，即中点。
Q: 为什么 fast 的起点有时是 head，有时是 head.next？
查看答疑 如果节点总数是偶数，有两个中点（如 1->2->3->4 的 2 和 3）。如果 fast = head, slow = head，结束时 slow 指向 3（右中点）。如果 fast = head.next, slow = head，相当于 fast 偷跑了一步，它会更早触底，此时 slow 会指向 2（左中点）。左中点在链表前半段断开时非常关键。

2. 为什么快慢指针判断环一定能相遇？而且一定是在一圈内？

直觉质疑 (Naive)

快指针走那么快，会不会直接就越过慢指针而永远错过相遇，无穷无尽地兜圈子下去？

数学证明 (Smart)

设此时 fast 在 slow 的后方，距离为 d（以顺时针计）。由于每经过一回合，fast 走 2，slow 走 1，它们之间的距离 $d$ 就必定缩短 1 步。距离是离散的整数，每次 -1，最终必然减为 0，也就是精确相遇，绝不存在“跳过”！整个追及过程必定发生在慢指针进入环内走完一圈之前。

💻 模板代码：反转链表与环检测

# 1. 迭代反转链表：经典三指针交响乐
def reverseList(head):
    prev = None
    curr = head
    while curr:
        nxt = curr.next    # 【存】先用 nxt 把下一个节点拉住，防止链子断裂掉入深渊
        curr.next = prev   # 【反】把当前节点的箭头反转指向前驱
        prev = curr        # 【推】prev 向前推一格，变成刚刚处理好的 curr
        curr = nxt         # 【推】curr 也安全地向前推一格，变成刚才拉住的 nxt
    return prev            # 最终 curr 掉出悬崖(None)，prev 站在原本的尾端(新的头)

# 2. 环检测并找到入环节点 (Floyd 判圈法)
def detectCycle(head):
    slow, fast = head, head
    # Phase 1: 判定是否有环
    while fast and fast.next:
        slow = slow.next
        fast = fast.next.next
        if slow == fast:
            # Phase 2: 找入口
            # 为什么重新放一个指针在 head 同步走就能相遇在入口？
            # 设 head 到入口长 a，入环到相遇点 b，环剩余 c。
            # fast = 2*slow => a + n(b+c) + b = 2(a+b) => a = (n-1)(b+c) + c
            # 说明 a 的长度恰好等于【绕 n-1 圈后再走 c 步的长度】！
            ptr = head
            while ptr != slow:
                ptr = ptr.next
                slow = slow.next
            return ptr
    return None
技巧时间空间为什么
反转链表O(N)O(1)只花了几个指针的额外空间，沿途修改所有的箭头。
找中点/找倒数第K个O(N)O(1)只遍历一趟即可到达指定相对位置，极大地避免了二次扫描。

🟢 典型例题变种比对

变种：K 个一组翻转链表

题解： 这要求我们不仅仅是全局 Reverse，而是要局部的。我们用一个 cur 不断往后探 $K$ 步，如果探不到 $K$ 个说明结束了，不翻转；如果探到了，记录下这个局部头以及这一段结束的 nxt 节点，切断后对这一小段执行标准的三指针 Reverse，翻转完后再把它和之前/之后的子链缝合起来。这极大地考验了 虚拟头节点 (Dummy Node) 的使用。没有 Dummy Node 处理这种可能改变原始头节点的问题极其痛苦！

🔴 易错点分析

❌ 丢失被覆盖的 Next

while curr:
    # 灾难：一上来就把 next 覆盖了
    curr.next = prev 
    prev = curr
    # 原本的 curr.next 彻底找不到了！
    curr = curr.next

✅ 正确的保存与反转

while curr:
    nxt = curr.next # 备份
    curr.next = prev
    prev = curr
    curr = nxt      # 恢复

🔷 竞赛拓展

跳表 (Skip List)： 链表虽然插入 O(1) 但查找 O(N)。如果有序的话如何 O(log N) 查找？可以仿造二分法建“快速通道”，给部分节点增加更高的“层数”（随机化分布），使得最顶层指针一口气能跳过一半的元素。这是 Redis ZSet 的核心底层数据结构。
